CompJoster

What it does

The CompJoster is a smart compost bin designed to resemble an arcade game. Users input their bannerID via a keypad, and the system detects and counts each container deposited. This data is uploaded to thecompjoster.co, where users can track their composting stats, view a campus-wide leaderboard, and earn points redeemable for prizes such as free cookies. The hardware ensures accurate detection and tracking of containers while gamifying composting to encourage participation.

How we built it

The prototype is constructed from cardboard with an attached 4x4 keypad connected to an Arduino breadboard for user input. Once a valid bannerID is entered, an ultrasonic sensor detects each container dropped, increments the total count, and associates it with the bannerID. This information is uploaded as a JSON file to our website, which parses the data and updates user stats and the leaderboard in real time.

Sorting Claw

Hardware and control systems project featuring a robotic sorting claw. The system uses two servo motors and one DC motor to control the claw’s movement in three-dimensional space. A 7.4 V Li-ion battery powers the system. To safely regulate voltage for the electronics, a buck converter is used. The motors are controlled by an Arduino Uno, with all embedded logic written in C++.

The claw is designed to pick up and sort coins based on their color using computer vision techniques implemented with OpenCV. It is mounted on a custom frame that allows motion along the X, Y, and Z axes. An external camera (either a laptop webcam or a Raspberry Pi camera) captures images of coins placed in front of the claw and streams them to the vision pipeline for real-time color detection.

This demo shows an early prototype of the system. Coins of different colors are presented to the camera, and OpenCV determines their color in real time. The detected color is then transmitted to the Arduino via serial communication. Based on the classification, the Arduino executes a predefined movement path: for example, if a red coin is detected, the claw moves to pick it up, closes to grasp the coin, rotates to the right, and releases it into the corresponding pile. If a brown coin is detected, the same sequence is performed, but the coin is deposited into the left pile instead.

Bleep Sleep

Hardware Components

Bleep Sleep consists of two main devices: the Brick and the Tracker. Both use Arduino R4 boards with ESP32-S3 microcontrollers to handle sensor input and wireless communication.

The Brick is a simple physical button that starts and stops sleep tracking. Pressing it before bed signals “I’m done for the day” and begins the sleep session. The Brick also has an ultrasonic sensor, lighting up when a hand is waved above it, reinforcing intentionality. It timestamps sleep sessions and acts as a reference for the Tracker’s data.

The Tracker collects biometric data to analyze sleep cycles. It includes a touch sensor (mocking BPM/SpO₂), a temperature sensor, and an EMG sensor for muscle activity detection. These sensors estimate sleep quality and patterns. Data is transmitted wirelessly to the web application for analysis.

Control and Integration

The Arduino boards coordinate the sensors, record timestamps, and communicate data to the backend in real-time. The Brick’s input triggers sleep tracking, while the Tracker continuously monitors physiological signals. This system demonstrates full hardware-software integration.

Demo Functionality

In the demo, the Brick and Tracker communicate with the web app to display real-time sleep metrics. Users can see session duration, estimated sleep cycles, and other biometric indicators, showing how the system functions in practice.

Smart Heating Mug

Project Overview

My team and I built a self-heating mug with an adjustable heating element and display to keep coffee warm in the comfortably drinkable range of 40–65º Celsius. While similar products exist, such as the Ember Mug 2, our version is a more affordable, all-day alternative with improved usability. Users can set the desired temperature using a rotary encoder, and both the set temperature and current mug temperature are displayed on an OLED screen.

Hardware Details

The hardware system includes a PCB-based control circuit, a MOSFET to regulate the heating element, and additional components for precise temperature measurement and control. The heating element is embedded inside the mug, and a temperature sensor constantly monitors the liquid temperature to ensure it remains within the user-specified range. The PCB integrates power regulation, sensor inputs, and display outputs, while the MOSFET allows for efficient switching of the heating element based on feedback from the sensor.

Demo Functionality

In the demo, the rotary encoder is used to set the desired temperature. Once set, the heating pads activate, and the liquid temperature begins to rise. By the end of the video, the OLED display shows an increase in temperature, demonstrating that the heating system effectively warms the liquid. Since the mug is a smart heating device, the primary functionality highlighted is its ability to heat on demand; cooling is simply a matter of turning the heating pads off and is less visually interesting.

Usage Assumptions

Several assumptions were made for the system’s use: the mug must remain plugged into a wall power source, the user can operate the rotary encoder without accessibility assistance, and the temperature sensor remains inside the mug at all times. It is also assumed that the user will adjust the temperature periodically to maintain the desired warmth, keeping the system active throughout the day.

SoftKiller

I spent the year implementing and benchmarking the SoftKiller pileup mitigation algorithm in Julia, reimplementing the C++ FastJet methods to improve runtime performance for high-pileup proton–proton collision events for CERN. Using the JuliaHEP/JetReconstruction.jl package and HepMC3 event data, I confirmed published results and expanded the repository with new methods, achieving up to 50% faster performance than the original C++ code at extreme pileup levels (μ = 1000).

To validate the implementation, I designed and ran benchmarking pipelines on the Brown CMS Tier-3 cluster, averaging over 100 trials per test and confirming reproducibility at high pileup. I also contributed to data processing workflows with ArgParse-configured pipelines, enabling flexible command-line simulations and jet clustering analysis. Findings from this work were presented at the BOOST 2025 international physics workshop to over 100 peers and experts.

This research demonstrates the combination of high-performance software optimization in Julia with rigorous computational physics validation, providing a robust framework for future high-energy collider analyses and real-time pileup mitigation studies.

Worm – FPGA

We set out to design an interactive game utilizing the HEX display of the FPGA board. The game, developed in Verilog using Quartus Prime, allows the user to control a perpetually moving HEX segment (lovingly named “The Worm”) that slithers between the 6 HEX displays.

Using the four KEY buttons on the FPGA board, the player can puppeteer The Worm in any of four directions. However, if The Worm exceeds the bounds of the HEX, the game ends and our pal, The Worm, “dies.”

The only way for this little detrivore to escape its doomed fate is to collect three pieces of fruit (blinking HEX segments) with the user’s guidance. One piece of fruit appears at the start of the game, and once collected, another appears in a different location. This repeats once more, and after collecting three fruits, The Worm earns victory.

In celebration, The Worm and 15 of his friends congregate in the three rightmost HEX displays and spell out a triumphant “YAY”.

BRICS

I spent my summer and fall semester contributing to the Brown Interaction Capture System (BRICS), a platform that captures and processes 4D spatial data (3D + time) using specialized cameras, compute systems, and AI models. My work focused on PCB prototyping.

I evaluated electronic components, considering electrical ratings, size, and system compatibility, enabling the successful assembly of 40 camera hardware prototypes for the project. Additionally, I designed and assembled 40+ PCBs, integrating Wi-Fi modules and ESP microcontrollers to support wireless control and real-time data transmission in these embedded systems.
